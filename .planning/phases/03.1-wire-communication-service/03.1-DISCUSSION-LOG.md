## Transport Architecture

| Option | Description | Selected |
|--------|-------------|----------|
| Channels-only | No HTTP relay. Simplest architecture. | |
| Channels-primary, relay fallback | Channels default, relay activates when Channels unavailable. | |
| Dual transport (both active) | Both Channels and HTTP relay always active. | |

**User's choice:** Neither option exactly — user described needing both transports plus batching/high-volume support and DuckDB write coordination. "We'll need both. Perhaps more than both."
**Notes:** User anticipates high-volume/high-density messages from sublimation stream. Also raised DuckDB single-writer constraint as a Wire responsibility — sessions need coordinated Ledger access.

---

## Write Coordination (follow-up from Transport)

| Option | Description | Selected |
|--------|-------------|----------|
| Wire coordinates writes | Wire includes write queue/scheduler. Sessions send write intents, Wire serializes to Ledger. | ✓ |
| Separate coordinator service | Dedicated write coordinator separate from Wire. | |
| Secondary owns all writes | Only Secondary writes to Ledger. No coordinator needed. | |

**User's choice:** Wire coordinates writes
**Notes:** Natural fit since Wire mediates all inter-session communication.

## Message Protocol Design

| Option | Description | Selected |
|--------|-------------|----------|
| Typed envelope | Defined schema: from, to, type, urgency, payload, timestamp, correlationId. Types as enum. | ✓ |
| Minimal envelope, opaque payload | Thin wrapper. Wire doesn't inspect payload. | |
| Channel-per-type | Each message type gets own channel. No envelope needed. | |

**User's choice:** Typed envelope
**Notes:** Enables routing, batching, and priority queuing by type.

## Relay Server Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Inline in Wire service | Wire's start() boots Bun.serve. Single process. | |
| Separate process via Conductor | Relay runs as separate Bun process. Conductor spawns and manages. | ✓ |
| Docker container via Conductor | Relay in Docker. | |

**User's choice:** Separate process via Conductor
**Notes:** Better isolation. Relay survives Wire restarts.

## Backpressure & Flow Control

| Option | Description | Selected |
|--------|-------------|----------|
| Urgency-based queue with limits | Separate queues per urgency. Urgent bypasses. Background bounded. | ✓ |
| Global rate limiter | Single queue with rate limiting. | |
| Producer-side throttling | Wire pushes backpressure signals to producers. | |

**User's choice:** Urgency-based queue with limits
**Notes:** Natural fit for Reverie's urgency model.

---

## Reconnection & Resilience

| Option | Description | Selected |
|--------|-------------|----------|
| Buffered reconnection with TTL | Wire buffers messages for disconnected sessions up to configurable TTL. | ✓ |
| Immediate failover | No buffering. Disconnection triggers session:lost immediately. | |
| Persistent message log | All messages persisted to Journal/Ledger. Full replay on reconnect. | |

**User's choice:** Buffered reconnection with TTL
**Notes:** Default ~30s TTL.

---

## Write Queue Semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Priority FIFO with batching | Priority queue by urgency. FIFO within priority. Batch compatible writes. Last-writer-wins with version check. | ✓ |
| Strict FIFO | All writes in arrival order. No priority. | |
| Session-authority model | Each session owns certain tables/namespaces. | |

**User's choice:** Priority FIFO with batching
**Notes:** Prevents background sublimation writes from blocking urgent recall-product writes.

---

## MCP SDK Integration Pattern

| Option | Description | Selected |
|--------|-------------|----------|
| Wire as MCP server with channel capability | MCP tools for session-to-session. Channels for event push. | |
| Wire as pure transport abstraction | Own API, MCP is implementation detail. | |
| Dual API surface | MCP tools for session-to-session AND native API for in-process consumers. | ✓ |

**User's choice:** Dual API surface
**Notes:** MCP for Claude Code sessions, native for Reverie module code. Maximum flexibility.
