# Phase 3.2: Assay Federated Search - Research

**Researched:** 2026-03-23
**Domain:** Federated search service across heterogeneous data providers (SQL + flat file)
**Confidence:** HIGH

## Summary

Assay is a thin federation layer that routes queries to registered data providers (Ledger and Journal), executes provider-specific optimized queries, and returns merged results annotated with provider metadata. It is NOT a search engine, indexer, or ranking system -- it is a query router and result merger. The complexity is deliberately minimal for M1: route queries, collect results, tag provenance.

The existing codebase provides every building block needed. Both Ledger and Journal already implement `query(criteria)` returning `Result<Array<{ id, data, ... }>>`. Switchboard provides the event emission pattern. The service factory pattern (`createContract` + options-based DI) is thoroughly validated across eight prior services. Assay's contribution is the capability declaration layer (providers declare what kinds of queries they support) and the federated dispatch loop (query all applicable providers in parallel, merge results with `_provider` metadata).

**Primary recommendation:** Build Assay as a standard Dynamo service following the exact same factory pattern as Wire, Magnet, and all prior services. Provider registration via Map, capability-based routing via a simple declaration object on each provider, parallel query dispatch via `Promise.allSettled`, and flat result array with per-provider summary diagnostics.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Options-based DI at init(). Providers injected via the options object, consistent with every other Dynamo service. Assay receives `{ ledger, journal, switchboard }` at init(). No auto-discovery, no registry lookup. Phase 4's framework (Armature) may add container-based resolution later, but Assay itself takes explicit provider references.
- **D-02:** Provider pool is extensible. Assay stores providers in an internal Map keyed by provider name. New providers can be registered after init via `registerProvider(name, provider)`. This supports Phase 4+ plugin providers without modifying Assay's core.
- **D-03:** Dual query API. Two entry points: `search(query)` (federated across all registered providers) and `searchProvider(name, query)` (targeted against a specific named provider).
- **D-04:** Query object shape: `{ criteria, providers?, options? }`. `criteria` is the plain object matching existing provider `query(criteria)` contract. `providers` is optional array to restrict federated search. `options` carries provider-specific hints (e.g., `{ sql: 'SELECT ...', limit: 50 }`).
- **D-05:** Provider capability declaration. Each provider declares its query capabilities. Ledger declares SQL capability. Journal declares frontmatter-scan capability. Assay routes the query using the most efficient method each provider supports.
- **D-06:** Capability-based routing. Assay inspects the query and each provider's declared capabilities to determine optimal query path. SQL hint + SQL-capable provider = direct SQL. Criteria keys = standard query(criteria). Unsupported query type = skip provider.
- **D-07:** Provider metadata on every result: `{ ...resultData, _provider: { name, type } }`.
- **D-08:** Provider-grouped result structure. Returns `{ results: [...], providers: { ledger: { count, timing }, journal: { count, timing } } }`. Flat result array with per-provider summary. Consumer (Reverie Mind) owns ranking -- Assay does not impose relevance ordering.
- **D-09:** No indexing in M1. Query federation only. Tika/Meilisearch/Opensearch are M2+ concerns.
- **D-10:** Switchboard events for all operations: `search:started`, `search:completed`, `search:error` with query shape, provider list, timing, result counts.

### Pattern 3: Federated Query Dispatch with Promise.allSettled
**What:** Parallel query dispatch to all eligible providers with individual failure isolation.
**When to use:** In the `search(query)` method.
**Example:**
```javascript
// Source: Derived from CONTEXT.md D-06/D-08

async function search(query) {
  const startTime = performance.now();

  // Determine which providers to query
  const targetProviders = query.providers
    ? [..._providers.entries()].filter(([name]) => query.providers.includes(name))
    : [..._providers.entries()];

  // Emit search:started
  if (_switchboard) {
    _switchboard.emit('search:started', {
      criteria: query.criteria,
      providers: targetProviders.map(([name]) => name),
    });
  }

  // Dispatch to each provider based on capabilities
  const promises = targetProviders.map(async ([name, { provider, capabilities }]) => {
    const providerStart = performance.now();
    let result;

    // Capability-based routing (D-06)
    if (query.options && query.options.sql && capabilities.supports.sql) {
      // Direct SQL path for SQL-capable providers
      result = await provider.query({ _raw_sql: query.options.sql, _limit: query.options.limit });
    } else if (capabilities.supports.criteria) {
      // Standard criteria path
      result = await provider.query(query.criteria);
    } else {
      // Provider cannot handle this query type -- skip
      return { name, results: [], timing: 0, skipped: true };
    }

    const timing = performance.now() - providerStart;

    if (!result.ok) {
      return { name, results: [], timing, error: result.error };
    }

    // Tag each result with provider metadata (D-07)
    const tagged = result.value.map(item => ({
      ...item,
      _provider: { name, type: capabilities.type }
    }));

    return { name, results: tagged, timing, count: tagged.length };
  });

  const settled = await Promise.allSettled(promises);
  // ... merge results, build providers summary, emit search:completed ...
}
```

### Pattern 4: Provider Metadata Tagging (D-07)
**What:** Every result carries `_provider: { name, type }` so consumers can route follow-up operations.
**When to use:** During result collection, before merging into flat array.
**Example:**
```javascript
// Source: CONTEXT.md D-07

// Each result from Journal gets:

```

### Anti-Patterns to Avoid
- **Lowest-common-denominator queries:** Do NOT force all providers through the same query path. D-06 explicitly requires capability-based routing. If Ledger supports SQL, pass the SQL directly. If Journal only supports criteria, use criteria.
- **Result ranking in Assay:** Do NOT add relevance scoring. Per D-08 and CONTEXT.md deferred ideas, ranking is Reverie's responsibility. Assay returns results in provider-insertion order.
- **Coupling to specific providers:** Do NOT reference Ledger or Journal internals. Assay interacts only through the DATA_PROVIDER_SHAPE contract (query method) plus the capability declaration object.
- **Synchronous iteration:** Do NOT query providers sequentially. Use `Promise.allSettled` for parallel dispatch -- both Ledger (async SQL) and Journal (async file reads) benefit from concurrent execution.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Service contract validation | Manual method checking | `createContract('assay', ASSAY_SHAPE, impl)` from `lib/contract.cjs` | Established pattern, catches wiring bugs at creation time |
| Result types | Custom error objects or exceptions | `ok()` / `err()` from `lib/result.cjs` | Uniform error handling across entire platform |
| Event emission | Custom EventEmitter or callback system | Switchboard `emit()` / `on()` | Platform standard, enables cross-service observability |

**Key insight:** Assay adds no new infrastructure dependencies. It composes existing pieces (provider query contract, Switchboard events, Result types, contract validation) into a federation pattern. The entire service can be implemented with zero npm dependencies and zero new lib/ additions.

## Common Pitfalls

### Pitfall 1: Swallowing Provider Errors Silently
**What goes wrong:** A provider's `query()` returns `Err`, but the federated search silently skips it without reporting, making the consumer think the search was comprehensive when it was partial.
**Why it happens:** Using a simple try/catch that logs nothing, or filtering out error results before returning.
**How to avoid:** Include provider errors in the `providers` summary of the response: `{ ledger: { count: 0, timing: 5, error: { code: 'QUERY_FAILED', message: '...' } } }`. Also emit `search:error` on Switchboard when any provider fails.
**Warning signs:** Tests that only verify happy paths. No test for partial-failure scenarios.

### Pitfall 2: Mutating Provider Results
**What goes wrong:** Adding `_provider` metadata by mutating the original result objects from Ledger/Journal, which are shared references from the provider's internal state.
**Why it happens:** Using direct assignment instead of spread/copy.
**How to avoid:** Always spread the result before adding metadata: `{ ...item, _provider: { name, type } }`. This creates a new object, leaving the provider's internal data untouched.
**Warning signs:** Tests that pass in isolation but fail when providers are reused across multiple searches.

### Pitfall 3: SQL Injection in Raw SQL Path
**What goes wrong:** The `options.sql` hint is passed directly to Ledger's query without validation, enabling unintended SQL.
**Why it happens:** Trusting that the caller (Reverie Mind) always sends safe SQL.
**How to avoid:** In M1, the raw SQL path is a trusted internal API -- only Reverie's Mind session calls it, not external users. Document this clearly. For M2, if Assay is exposed via Pulley CLI/MCP, add query validation. For now, the capability declaration system already prevents SQL from reaching non-SQL providers (Journal would skip it), which is the primary safety concern.
**Warning signs:** External exposure of the search API without input validation.

### Pitfall 4: Blocking on Slow Providers
**What goes wrong:** If Journal has thousands of markdown files, `query(criteria)` scans them all synchronously, blocking the entire federated search.
**Why it happens:** Journal's current implementation reads all `.md` files sequentially in `query()`.
**How to avoid:** Use `Promise.allSettled` for parallel dispatch. Even though Journal internally reads files sequentially, Ledger's SQL query can complete in parallel. If a provider is slow, other providers' results arrive independently. The `timing` field in the response makes slowness visible. Consider adding a query timeout option in the future.
**Warning signs:** Federated search timing dominated by the slowest provider even when faster providers have results ready.

### Pitfall 5: Forgetting Lifecycle Methods
**What goes wrong:** Assay's `init()` accepts providers but forgets to validate they implement the query contract, or `stop()` does not clean up internal state.
**Why it happens:** Assay is a thin layer, so developers assume lifecycle is trivial.
**How to avoid:** Follow the established pattern exactly: init validates options, start sets _started flag, stop clears state, healthCheck reports status. Test each lifecycle method explicitly.
**Warning signs:** healthCheck returning stale status after stop(), or init accepting a provider without a query method.

## Code Examples

### Example 1: Complete createAssay Factory Skeleton
```javascript
// Source: Derived from established wire.cjs/magnet.cjs patterns + CONTEXT.md decisions
'use strict';

const { ok, err, createContract } = require('../../../lib/index.cjs');

const ASSAY_SHAPE = {
  required: ['init', 'start', 'stop', 'healthCheck', 'search', 'searchProvider', 'registerProvider'],
  optional: ['getProviders']
};

function createAssay() {
  let _started = false;
  let _switchboard = null;
  const _providers = new Map();

  function init(options) {
    if (!options) {
      return err('INIT_FAILED', 'Assay init requires options object');
    }
    _switchboard = options.switchboard || null;

    // Register initial providers from options (D-01)
    if (options.ledger) {
      _providers.set('ledger', {
        provider: options.ledger,
        capabilities: { name: 'ledger', type: 'sql', supports: { sql: true, criteria: true } }
      });
    }
    if (options.journal) {
      _providers.set('journal', {
        provider: options.journal,
        capabilities: { name: 'journal', type: 'frontmatter', supports: { sql: false, criteria: true } }
      });
    }

    return ok(undefined);
  }

  // ... search, searchProvider, registerProvider, lifecycle methods ...

  return createContract('assay', ASSAY_SHAPE, { init, start, stop, healthCheck, search, searchProvider, registerProvider });
}

module.exports = { createAssay, ASSAY_SHAPE };
```

## Open Questions

1. **Raw SQL pass-through to Ledger**
   - What we know: D-06 says "If the query includes `options.sql` and the provider supports SQL -> pass the SQL directly." Ledger's current `query(criteria)` method only accepts criteria objects -- it does not accept raw SQL strings.
   - What's unclear: Should Assay call Ledger's query() with a special criteria key (e.g., `{ _raw_sql: 'SELECT ...' }`), or should Ledger expose a separate `executeSql()` method, or should Assay use Ledger's backend directly?
   - Recommendation: Use a special criteria key convention (`_raw_sql`). This keeps Assay interacting through the existing `query(criteria)` contract without adding new methods. Ledger's `query()` already handles special keys (`_limit`). Adding `_raw_sql` detection is a small backward-compatible change to Ledger, or Assay can perform the SQL translation itself by wrapping the criteria into the SQL WHERE clause format that Ledger expects. For M1, the standard `query(criteria)` path is sufficient -- raw SQL optimization can be deferred until Reverie's Mind actually needs it.

2. **Provider capability schema extensibility**
   - What we know: M1 needs `sql` and `criteria` capability flags. M2 may add `fulltext`, `vector`, etc.
   - What's unclear: Should the capability schema be a flat boolean map (`supports: { sql: true }`) or a richer structure with metadata?
   - Recommendation: Start with flat boolean map. It is the simplest thing that works for M1. When M2 adds new capability types, extend the `supports` object. No schema migration needed -- new keys are additive.

3. **Parallel vs sequential query dispatch timing accuracy**
   - What we know: `Promise.allSettled` runs providers in parallel. Per-provider timing uses `performance.now()` around each provider call.
   - What's unclear: If providers share resources (both hit disk), does parallel execution inflate per-provider timing measurements?
   - Recommendation: Measure wall-clock time per provider. It is what consumers care about. If profiling shows contention, document it in the providers summary. This is a diagnostic concern, not a correctness concern.
