# Phase 12.1: Platform Launch Readiness - Research

**Researched:** 2026-03-25
**Domain:** Module discovery, Claude Code skill integration, multi-triplet session isolation, E2E validation
**Confidence:** HIGH

## Summary

Phase 12.1 bridges the gap between a codebase where every component exists and works in isolation (tests pass) and a platform that actually runs end-to-end. The six success criteria span five domains: (1) automatic module discovery and registration during bootstrap, (2) Claude Code skill files as user-facing entry points, (3) session triplet spawning with visual markers, (4) multi-triplet concurrency isolation in Wire and Switchboard, and (5) scripted E2E validation of the full lifecycle.

The codebase is remarkably well-positioned for this phase. Circuit's `registerModule()` works but is never called from bootstrap -- a single function addition to `core.cjs` closes this gap. Exciter has `registerSkill()` in its optional contract but unimplemented -- implementing it writes `.md` files to `.claude/skills/` following Claude Code's skill discovery convention. Wire registry, Switchboard, and Session Manager all exist with complete APIs; multi-triplet isolation requires extending session IDs with a triplet prefix namespace and adding payload-level filtering. The E2E validation layer is net-new but orchestrates existing components.

**Primary recommendation:** Build from the inside out -- module discovery first (prerequisite for everything), then skills (user-facing surface), then multi-triplet isolation (Wire/Switchboard extension), then visual markers (output concern), then E2E validation (proves it all works).

### Pattern 1: Module Discovery (parallels Plugin Discovery)

**What:** Scan `modules/` directory for subdirectories containing a manifest (module.json or exported manifest.cjs). Load, validate, and register via Circuit.
**When to use:** During bootstrap, after services/providers are booted and Circuit is created.
**Reference:** `core/armature/plugin.cjs` discoverPlugins() pattern -- identical directory scan + manifest validation + register flow.

```javascript
// Source: core/armature/plugin.cjs pattern adapted for modules
function discoverModules(modulesDir) {
  if (!fs.existsSync(modulesDir)) {
    return [];
  }
  const entries = fs.readdirSync(modulesDir, { withFileTypes: true });
  const modulePaths = [];
  for (const entry of entries) {
    if (!entry.isDirectory()) continue;
    const moduleDir = path.join(modulesDir, entry.name);
    // Check for manifest.cjs (CJS module manifest) or module.json
    const manifestCjs = path.join(moduleDir, 'manifest.cjs');
    const moduleJson = path.join(moduleDir, 'module.json');
    if (fs.existsSync(manifestCjs) || fs.existsSync(moduleJson)) {
      modulePaths.push(moduleDir);
    }
  }
  return modulePaths;
}
```

The bootstrap integration in core.cjs adds module discovery between step 7.5c (platform commands) and step 8 (return success):

```javascript
// 7.5d. Discover and register modules
const modulesDir = paths.root + '/modules';
const moduleDirs = discoverModules(modulesDir);
for (const moduleDir of moduleDirs) {
  // Load manifest
  const manifest = require(path.join(moduleDir, 'manifest.cjs'));
  const moduleManifest = manifest.REVERIE_MANIFEST || manifest.default || manifest;
  if (!moduleManifest.enabled) continue;
  // Load entry point and register
  const entry = require(path.join(moduleDir, moduleManifest.main));
  circuit.registerModule(moduleManifest, entry.register);
}
```

### Pattern 3: Triplet ID Namespace for Wire Isolation

**What:** Each triplet gets a unique prefix (e.g., `triplet-a1b2`). All Wire session IDs within that triplet are prefixed: `triplet-a1b2:primary`, `triplet-a1b2:secondary`, `triplet-a1b2:tertiary`. Wire registry, topology, and subscribers all operate within this namespace.
**When to use:** When Session Manager creates a new triplet.

```javascript
// Source: D-06 decision, extending session-manager.cjs pattern
const crypto = require('node:crypto');

function generateTripletId() {
  const hash = crypto.randomBytes(2).toString('hex'); // 4-char hex: a1b2
  return 'triplet-' + hash;
}

function makeTripletSessionId(tripletId, identity) {
  return tripletId + ':' + identity; // e.g., 'triplet-a1b2:primary'
}

function extractTripletId(sessionId) {
  const idx = sessionId.indexOf(':');
  return idx >= 0 ? sessionId.slice(0, idx) : null;
}
```

### Pattern 4: Visual Session Markers

**What:** Terminal title escape sequences + ANSI color-coded output prefixes applied per-session.
**When to use:** On session spawn, set terminal title and configure output prefix for each session role.

```javascript
// Terminal title: \033]0;Title\007
function setTerminalTitle(role, shortHash) {
  const title = 'Dynamo ' + role + ' ' + shortHash;
  process.stdout.write('\x1b]0;' + title + '\x07');
}

// Color-coded prefixes using ANSI 256-color that work on dark + light terminals
// Recommendation: semantic colors that remain readable on both backgrounds
const ROLE_COLORS = Object.freeze({
  Face: '\x1b[38;5;39m',        // Bright blue -- user-facing, trustworthy
  Mind: '\x1b[38;5;214m',       // Amber/orange -- cognitive, warm
  Subconscious: '\x1b[38;5;141m', // Soft purple -- subliminal, ethereal
});
const RESET = '\x1b[0m';

function formatPrefix(role, shortHash) {
  const color = ROLE_COLORS[role] || '';
  return color + '[' + role + ' ' + shortHash + ']' + RESET;
}
```

### Pattern 5: Subscriber-Level Triplet Filtering (Switchboard)

**What:** Per D-07, Switchboard remains a global event bus. Triplet filtering happens at the subscriber level. Events carry `triplet_id` in their payload; listeners check it.
**When to use:** Whenever a module subscribes to Switchboard events that are triplet-scoped.

```javascript
// Helper: create a triplet-scoped listener
function createTripletListener(switchboard, eventName, tripletId, handler) {
  return switchboard.on(eventName, function (payload, eventName) {
    if (payload && payload.triplet_id === tripletId) {
      handler(payload, eventName);
    }
  });
}
```

### Anti-Patterns to Avoid

- **Modifying Switchboard core for triplet scoping:** D-07 explicitly says Switchboard remains a global bus. Do NOT add triplet-aware routing inside Switchboard. Filtering is at the subscriber level.
- **Hardcoding module paths:** Module discovery must scan the directory, not hardcode `require('./modules/reverie/reverie.cjs')`. The whole point of INT-01/INT-03 is automatic discovery.
- **Skills calling raw APIs:** Skills are conversational wrappers (D-03). They should present results in Claude's response style, not dump CLI output. But internally they invoke the same Pulley commands.
- **Blocking bootstrap on module registration failure:** Module registration should be isolated -- a module failure should log an error but not crash the platform (same as plugin isolation pattern in lifecycle.cjs).
- **Creating skill files at require-time:** Skills must be written during Exciter's start() or module registration phase, not at import time. File I/O at require-time breaks testability.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Module manifest validation | Custom validation | lib/schema.cjs validate() + MODULE_MANIFEST_SCHEMA | Already exists in circuit/module-manifest.cjs |
| Directory scanning | Recursive glob | fs.readdirSync + isDirectory (plugin.cjs pattern) | Simple, no recursion needed, zero deps |
| Session ID uniqueness | UUID library | crypto.randomBytes(2).toString('hex') | 4-char hex gives 65K namespace, sufficient for triplet IDs |
| YAML frontmatter generation | YAML parser library | String template concatenation | Frontmatter is simple key-value, no nested structures |
| Terminal color codes | chalk/kleur npm | Raw ANSI escape sequences | Zero-dependency, 3 colors total |
| JSON checkpoint log | Custom logger | JSON.stringify + Bun.write | Structured JSON is trivially serializable |

**Key insight:** This phase is an integration and wiring phase, not a feature-building phase. Nearly every component already exists. The work is connecting them correctly, adding the thin discovery/skill layers, and proving the connections work end-to-end.

## Common Pitfalls

### Pitfall 1: Module Discovery Order vs. Service Availability
**What goes wrong:** Modules are discovered before all services are booted, causing getService() to return null facades.
**Why it happens:** Bootstrap calls module discovery too early in the boot sequence.
**How to avoid:** Module discovery MUST happen AFTER lifecycle.boot() completes and all facades are available. In core.cjs, this is after step 7 (boot) and after step 7.5b (circuit creation). The current code structure supports this -- insert discovery between 7.5c and 8.
**Warning signs:** `SERVICE_NOT_FOUND` or `PROVIDER_NOT_FOUND` errors during module registration.

### Pitfall 2: Exciter Service Dependency on registerSkill()
**What goes wrong:** registerSkill() needs Lathe to write files, but Exciter doesn't have Lathe reference when modules call registerSkill() during registration.
**Why it happens:** Exciter.init() stores Lathe reference, but registerSkill() might be called before start().
**How to avoid:** Exciter.init() already stores `_lathe`. registerSkill() should use `_lathe` (available after init). Modules register during bootstrap, which happens after Exciter.init(). Verify init order.
**Warning signs:** `_lathe is null` errors when writing skill files.

### Pitfall 3: Wire Registry Session ID Collision Across Triplets
**What goes wrong:** Two triplets spawn sessions with the same identity name (e.g., both have a 'secondary'). Without triplet prefix, Wire registry overwrites the first session's entry.
**Why it happens:** Current session IDs use `reverie-secondary-{timestamp}` which is globally unique by timestamp but doesn't namespace per-triplet. With multi-triplet, the identity-based routing breaks.
**How to avoid:** D-06 solves this: all session IDs use the triplet prefix (`triplet-a1b2:secondary`). Wire topology's validateRoute uses identity extracted from the prefixed session ID. Ensure ALL Wire operations (register, subscribe, send, unregister) use the prefixed ID.
**Warning signs:** Messages arriving at the wrong triplet's session. Wire topology violations between sessions that should be connected.

### Pitfall 4: Skill Files Written But Not Discovered
**What goes wrong:** Skills are written to `.claude/skills/` but Claude Code doesn't see them because the directory doesn't exist, permissions are wrong, or the SKILL.md format is invalid.
**Why it happens:** Claude Code discovers skills at startup; skills written after session start require session restart to pick up. Also, YAML frontmatter errors (missing `---` delimiters) cause silent failure.
**How to avoid:** (1) Ensure `.claude/skills/` directory exists before writing (mkdir -p pattern). (2) Validate SKILL.md format before writing -- both `---` delimiters must be present. (3) Skills should be written during platform bootstrap, not dynamically during a session. (4) Document that a session restart may be needed after first install.
**Warning signs:** `/dynamo` or `/reverie` commands not appearing in Claude Code's skill list.

### Pitfall 5: Switchboard Event Payload Missing triplet_id
**What goes wrong:** An event fires without triplet_id in the payload. All triplet-scoped listeners ignore it, causing lost events.
**Why it happens:** Not all Switchboard emit calls are updated to include triplet_id.
**How to avoid:** Create a helper function that wraps Switchboard.emit to automatically inject triplet_id from the current Session Manager context. Use it consistently across all triplet-scoped emissions.
**Warning signs:** Events fire but no listener responds. Debugging shows payload lacks triplet_id.

### Pitfall 6: Auto-Init Race Condition
**What goes wrong:** Two skills invoke concurrently, both detect Dynamo isn't booted, both attempt to bootstrap simultaneously.
**Why it happens:** D-04 requires auto-init behavior. If two skills trigger at the same time, there's no singleton guard.
**How to avoid:** Use a singleton bootstrap promise. First call creates the promise, subsequent calls await the same promise. Store in a module-level variable.
**Warning signs:** Double-initialization errors, duplicate service registrations.

### Pitfall 7: E2E Validation Assumes Claude Code Session Availability
**What goes wrong:** The scripted live session (D-14) tries to spawn Secondary/Tertiary sessions but the test environment doesn't have Claude Code available or Channels API access.
**Why it happens:** E2E validation with real sessions requires an active Claude Code session with Channels enabled.
**How to avoid:** D-13 mandates two layers: integration tests (mock everything, fast/repeatable) AND scripted live session (real Claude Code). Integration tests can run anywhere. Live session is documented as requiring an active Claude Code Max session.
**Warning signs:** Session spawn returns DOCKER_UNAVAILABLE or process exits immediately.

### Bootstrap Module Discovery Integration

```javascript
// Source: core/core.cjs integration point (between 7.5c and 8)
// Based on plugin.cjs discoverPlugins pattern

const { discoverModules } = require('./armature/module-discovery.cjs');

// In bootstrap(), after circuit creation:

// 7.5d. Discover and register modules
const modulesDir = options.modulesDir || paths.root + '/modules';
const moduleResults = [];
const moduleDirs = discoverModules(modulesDir);

for (const moduleDir of moduleDirs) {
  try {
    // Load manifest (supports manifest.cjs export pattern)
    const manifestModule = require(path.join(moduleDir, 'manifest.cjs'));
    const manifest = manifestModule.REVERIE_MANIFEST || manifestModule;

    if (manifest.enabled === false) {
      moduleResults.push({ name: manifest.name, status: 'disabled' });
      continue;
    }

    // Load entry point
    const entry = require(path.join(moduleDir, manifest.main));

    // Register via Circuit
    const result = circuit.registerModule(manifest, entry.register);
    moduleResults.push({ name: manifest.name, status: result.ok ? 'registered' : 'error' });
  } catch (e) {
    moduleResults.push({ name: path.basename(moduleDir), status: 'error', error: e.message });
    // Module failure is isolated -- platform continues
  }
}
```

### Exciter registerSkill() Implementation

```javascript
// Source: D-05 decision + Claude Code skill docs
// Added to core/services/exciter/exciter.cjs

function registerSkill(skillName, options = {}) {
  if (!_lathe) {
    return err('NOT_INITIALIZED', 'Exciter must be initialized before registering skills');
  }

  const skillDir = path.join(_projectRoot, '.claude', 'skills', skillName);
  const skillPath = path.join(skillDir, 'SKILL.md');

  // Build YAML frontmatter
  const frontmatter = [
    '---',
    'name: ' + skillName,
  ];
  if (options.description) {
    frontmatter.push('description: ' + options.description);
  }
  if (options.disableModelInvocation) {
    frontmatter.push('disable-model-invocation: true');
  }
  if (options.allowedTools) {
    frontmatter.push('allowed-tools: ' + options.allowedTools);
  }
  frontmatter.push('---');

  const content = frontmatter.join('\n') + '\n\n' + (options.content || '');

  // Ensure directory exists
  _lathe.mkdirSync(skillDir, { recursive: true });

  // Write SKILL.md
  return _lathe.writeFileSync(skillPath, content);
}
```

## Open Questions

1. **Skill content execution model**
   - What we know: Skills are markdown instructions that Claude reads and follows. They are NOT executable code. Claude interprets the instructions and uses its tools to fulfill them.
   - What's unclear: How `/dynamo validate` orchestrates a real live session from within a skill context. The skill would need to instruct Claude to use Bash tool to run validation scripts.
   - Recommendation: `/dynamo validate` skill instructs Claude to execute `bun run modules/reverie/validation/live-session-script.cjs` via Bash, then presents results conversationally. The skill is the conversational wrapper; the script is the engine.

2. **Auto-init default recommendation**
   - What we know: D-04 says auto-init is configurable. Default is Claude's discretion.
   - Recommendation: Default to auto-boot (`auto_init: true`). Rationale: skills are invoked when the user wants to interact with Dynamo/Reverie. Requiring explicit init creates friction. The config key lets power users disable it.

3. **Checkpoint log format recommendation**
   - What we know: D-15 says structured checkpoint log with pass/fail per criterion.
