# Phase 12.1: Platform Launch Readiness - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Close every gap between "tests pass" and "the platform runs" — module discovery and loading, Claude Code hook wiring, session triplet spawning and management via Claude skills, multi-triplet concurrency, visual session distinction, and full end-to-end system validation. Claude skills serve as the user-facing entry points for initializing and managing Dynamo and Reverie. This phase delivers: automatic module discovery from `modules/` directory with Circuit registration on boot (INT-01 complete wiring), Claude skills (`/dynamo`, `/reverie`) as conversational entry points for platform and session management (INT-02 extension, SES-01/SES-02/SES-03 user surface), multi-triplet concurrency with Wire registry isolation and Switchboard event scoping (INT-03 runtime validation), visual session distinction via terminal titles and color-coded role prefixes, and scripted E2E validation proving the full lifecycle works end-to-end.

</domain>

<decisions>
## Implementation Decisions

### Skill Design
- **D-01:** `/dynamo` is a platform dashboard skill — shows platform health, loaded modules, active services, hook status. Single skill, conversational presentation. Not a namespace for subcommands.
- **D-02:** `/reverie` is a session management hub — shows current mode (Active/Passive/Dormant), session topology, and offers actions: start session, view status, inspect memory, stop. Interactive entry point for everything Reverie.
- **D-03:** Skills are conversational wrappers over existing Pulley CLI commands. They call the same underlying logic but present results contextually in Claude's response. CLI stays for scripting/automation. Skills are the friendly human interface.
- **D-04:** Auto-init behavior is configurable in Dynamo config. If Dynamo hasn't booted when a skill is invoked, behavior depends on config: auto-boot transparently, or require explicit init. Default behavior is Claude's discretion. The config key must exist.
- **D-05:** Skills are registered via Exciter's `registerSkill()` method (currently stubbed as optional). Phase 12.1 implements this. Skills are `.md` files written to `.claude/skills/` following Claude Code's skill discovery convention.

### Multi-Triplet Isolation
- **D-06:** Triplet ID prefix for Wire session isolation. Each triplet gets a unique ID (e.g., `triplet-a1b2`). Session IDs become `triplet-a1b2:primary`, `triplet-a1b2:secondary`, `triplet-a1b2:tertiary`. Wire registry partitions routing by triplet prefix. Clean namespace isolation.
- **D-07:** Switchboard events carry triplet_id in payload. Listeners filter by triplet. Scoping is at the subscriber level — Switchboard itself remains a global event bus. Minimal Switchboard changes required.
- **D-08:** DuckDB single-writer constraint stays. Each triplet's Secondary writes its own data. Write coordinator serializes cross-triplet writes. Triplet ID column added to relevant Ledger tables for data isolation.
- **D-09:** Configurable concurrent triplet cap, default 3 (9 total sessions). Enforced at Session Manager level. Set via Dynamo config (`max_triplets`).

### Visual Session Markers
- **D-10:** Dual visual distinction: terminal title (`\033]0;Dynamo Face a1b2\007`) AND color-coded output prefix (`[Face a1b2]`, `[Mind a1b2]`, `[Subconscious a1b2]`). Both are applied. Terminal title for window/tab identification, prefix for in-response identification.
- **D-11:** Color scheme is Claude's discretion — pick colors that work well across common terminal themes (both dark and light backgrounds). Semantic mapping: distinct colors for Face, Mind, Subconscious roles.
- **D-12:** Short hash suffix from triplet ID visible in markers: `[Face a1b2]`, `[Mind a1b2]`. Users can tell which sessions belong together when multiple triplets are running.

### E2E Validation
- **D-13:** Two validation layers: integration test harness (bun:test, mocks Claude Code hook contract, fast/repeatable/CI-friendly) PLUS scripted live session (boots Dynamo in real Claude Code, verifies each success criterion in sequence against real sessions).
- **D-14:** Scripted live session boots Secondary and Tertiary sessions automatically, exactly as would happen when the user invokes the `/reverie` skill to start a triplet session. Full lifecycle: boot -> module loads -> hooks fire -> personality injects -> formation fires -> recall works -> REM runs.
- **D-15:** Validation results written to a structured checkpoint log file (JSON or markdown) with pass/fail per success criterion, timestamps, and evidence. Human-reviewable artifact persisted after validation.
- **D-16:** E2E validation runnable as `/dynamo validate` Claude skill. Users can run it anytime to verify their installation works end-to-end. Conversational presentation of results.
- **D-17:** Go-live gate requires all 6 roadmap success criteria green. Any red blocks go-live. Yellow (degraded but functional) is acceptable with documented workaround.

### Claude's Discretion
- Module discovery mechanism details (follow plugin.cjs discoverPlugins() pattern or different approach)
- Auto-init default behavior (whether to auto-boot or require explicit init by default)
- Color scheme selection for session role prefixes (must work on dark and light terminal themes)
- Integration test harness structure and mock contract design
- Checkpoint log file format details (JSON vs markdown, exact schema)
- Skill `.md` file structure and content for `/dynamo` and `/reverie`
- How `/dynamo validate` orchestrates the scripted live session within a Claude skill context

</decisions>

<specifics>
## Specific Ideas

### Skills as the Human Interface
The user chose skills as conversational wrappers — not CLI mirrors, not replacements. `/dynamo` is a dashboard, `/reverie` is a session management hub. They present the same information as CLI commands but in a way that fits Claude's conversational context. This means skills can be smarter — `/reverie` can show "You have an active triplet, would you like to inspect it?" rather than just dumping status output.

### Configurable Auto-Init
The user specifically requested that auto-initialization behavior be toggleable in Dynamo config rather than picking one default. This means the config infrastructure must include an `auto_init` key, and skills must check it before deciding whether to bootstrap on first invocation.

### Triplet ID Prefix as Namespace
The prefix pattern (`triplet-a1b2:primary`) is clean — it extends Wire's existing session ID concept without changing the fundamental routing model. Topology rules apply within the prefix namespace. Cross-triplet communication is not needed (and would be a new capability for a future phase).

### Dual Visual Distinction
Both terminal title AND color-coded output prefix. The user wants both layers. Terminal titles help when managing multiple windows/tabs; color-coded prefixes help when reading interleaved output in a single view. The short hash suffix ties sessions to their triplet visually.

### E2E as Skill + Test + Live
The validation strategy is defense-in-depth: integration tests catch regressions fast, the scripted live session proves real Claude Code integration, and the `/dynamo validate` skill lets users verify their own installation anytime. The scripted session must boot Secondary and Tertiary exactly as `/reverie` would — no shortcuts, no mocks.

</specifics>
