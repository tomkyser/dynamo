# Phase 9.1: Claude Code Integration Layer - Research

**Researched:** 2026-03-24
**Domain:** Claude Code integration surface (transcript JSONL I/O, hook/agent/skill/settings management, CLAUDE.md section ownership)
**Confidence:** HIGH

## Summary

Phase 9.1 builds two new platform components: **Lithograph** (a provider for reading, querying, and atomically manipulating Claude Code transcript JSONL files) and **Exciter** (a service that owns the entire Claude Code integration surface -- hook registration, agent definitions, skill definitions, settings.json at project and user scope, and CLAUDE.md section management). These components centralize Claude Code integration behind Dynamo's contract model, enforcing the core value that everything routes through Dynamo.

The transcript manipulation mechanism has been verified via source code analysis of the claude-mem plugin's "endless mode" branch. Claude Code passes `transcript_path` to hook handlers via stdin JSON. The transcript is a read/write JSONL file where each line is a conversation event. Direct manipulation (replacing `block.input` with `{}` to clear tool inputs) and `additionalContext` injection are both proven techniques. Lithograph wraps this undocumented interface behind a versioned parser with defensive format detection.

Exciter acts as the architectural enforcement point for Claude Code features. Without it, each module would independently manage settings.json, agent files, and CLAUDE.md -- the exact "bypass the patterns and paths Dynamo defines" anti-pattern. Exciter delegates hook registry internals to Armature's existing `createHookRegistry()` while owning the registration facade that modules call. Reverie's existing 10-line direct registry call in `reverie.cjs` collapses to 1 Exciter call, proving the contract.

**Primary recommendation:** Build Lithograph as a DATA_PROVIDER_SHAPE-conforming provider with sync reads and async atomic writes. Build Exciter as a service with init/start/stop lifecycle. Migrate Reverie's hook registration in the same phase as the validation gate.

### Pattern 1: Provider Contract Conformance (Lithograph)
**What:** Lithograph implements DATA_PROVIDER_SHAPE exactly as Journal and Ledger do: `init`, `start`, `stop`, `healthCheck`, `read`, `write`, `query`, `delete`. Validated via `validateDataProvider()`.
**When to use:** Always -- this is the provider contract.
**Example:**
```javascript
// Source: core/providers/provider-contract.cjs + core/providers/journal/journal.cjs
'use strict';
const { ok, err } = require('../../../lib/index.cjs');
const { validateDataProvider } = require('../provider-contract.cjs');

function createLithograph() {
  let _transcriptPath = null;
  let _started = false;
  let _lathe = null;
  let _cachedLines = null; // Parsed JSONL lines cache

  function init(options) {
    // No transcript_path at init -- injected per-session via setTranscriptPath()
    if (!options || !options.lathe) {
      return err('INIT_FAILED', 'Lithograph init requires lathe service');
    }
    _lathe = options.lathe;
    return ok(undefined);
  }

  // Session-scoped injection (D-03)
  function setTranscriptPath(transcriptPath) {
    _transcriptPath = transcriptPath;
    _cachedLines = null; // Invalidate cache
    return ok(undefined);
  }

  // Sync read (D-04) -- reads and parses the full JSONL file
  function read(id) { /* ... */ }

  // Async write with atomic semantics (D-04)
  async function write(id, data) { /* ... */ }

  // Query by criteria: { type, tool_use_id, turn_type, time_range }
  function query(criteria) { /* ... */ }

  function del(id) { /* ... */ }

  // ... lifecycle methods ...

  const extra = { setTranscriptPath };
  return validateDataProvider('lithograph', {
    init, start, stop, healthCheck, read, write, query, delete: del,
    ...extra,
  });
}
```

### Pattern 2: Service with Facade Delegation (Exciter)
**What:** Exciter is a service that owns the Claude Code integration surface. It delegates to Armature's hook registry for the mechanics but owns the registration contract that modules call.
**When to use:** For all Claude Code feature management.
**Example:**
```javascript
// Source: core/armature/hooks.cjs pattern + CONTEXT.md D-05/D-09
'use strict';
const { ok, err, createContract } = require('../../../lib/index.cjs');
const { createHookRegistry } = require('../../armature/hooks.cjs');

const EXCITER_SHAPE = {
  required: ['init', 'start', 'stop', 'healthCheck',
             'registerHooks', 'getRegisteredHooks',
             'installAgent', 'removeAgent', 'listAgents',
             'updateSettings', 'readSettings',
             'claimSection', 'updateSection', 'releaseSection'],
  optional: ['registerSkill'],
};

function createExciter() {
  let _hookRegistry = null;
  let _switchboard = null;
  let _lathe = null;
  // ... internal state ...

  function init(options) {
    _switchboard = options.switchboard;
    _lathe = options.lathe;
    _hookRegistry = createHookRegistry();
    return ok(undefined);
  }

  // D-09: Facade method for hook registration
  function registerHooks(moduleName, handlers) {
    // handlers: { SessionStart: fn, UserPromptSubmit: fn, ... }
    let count = 0;
    for (const [hookType, handler] of Object.entries(handlers)) {
      const result = _hookRegistry.register(hookType, moduleName, handler);
      if (!result.ok) return result;
      count++;
    }
    // Wire to Switchboard after registration
    _hookRegistry.wireToSwitchboard(_switchboard);
    return ok(count);
  }
  // ... other methods ...

  return createContract('exciter', EXCITER_SHAPE, impl);
}
```

### Pattern 3: Versioned Parser for Undocumented Format (Lithograph Parser)
**What:** The transcript JSONL format is undocumented and may change. A versioned parser detects the format on first read and selects the appropriate parser.
**When to use:** Lithograph's internal parsing of transcript files.
**Example:**
```javascript
// Source: Research finding from claude-mem analysis
'use strict';

// Parser registry -- add new versions as format changes
const PARSERS = {
  v1: {
    detect: (firstLine) => {
      // v1: each line is a JSON object with role/content/tool_use fields
      const parsed = JSON.parse(firstLine);
      return parsed.role !== undefined || parsed.type !== undefined;
    },
    parseLine: (line) => {
      const obj = JSON.parse(line);
      return { ...obj, _parserVersion: 'v1' };
    },
  },
  // Future: v2, v3 as format changes
};

function detectVersion(firstLine) {
  for (const [version, parser] of Object.entries(PARSERS)) {
    try {
      if (parser.detect(firstLine)) return version;
    } catch (_) { continue; }
  }
  return null; // Unknown format
}

function parseTranscript(content) {
  const lines = content.trim().split('\n').filter(Boolean);
  if (lines.length === 0) return { version: null, entries: [] };

  const version = detectVersion(lines[0]);
  if (!version) return { version: null, entries: [], error: 'UNKNOWN_FORMAT' };

  const parser = PARSERS[version];
  const entries = [];
  for (const line of lines) {
    try {
      entries.push(parser.parseLine(line));
    } catch (_) {
      // Skip malformed lines -- defensive, not paranoid
    }
  }
  return { version, entries };
}
```

### Pattern 5: Settings.json Merge with Scope Precedence
**What:** Exciter manages settings.json at both project (`.claude/settings.json`) and user (`~/.claude/settings.json`) scopes. Writes are scoped; reads can merge with precedence.
**When to use:** Hook registration to settings.json, permission management.
**Example:**

```javascript
// Claude Code precedence (from official docs):
// 1. Managed policy settings (enterprise, highest priority)
// 2. User settings: ~/.claude/settings.json
// 3. Project settings: .claude/settings.json
// 4. Local settings: .claude/settings.local.json (lowest, not shareable)
// All levels merge -- hooks from all levels active unless disableAllHooks
```

### Anti-Patterns to Avoid
- **Lithograph caching stale data:** The transcript file grows during a session. Never cache the entire file indefinitely. Cache parse results but invalidate when the file is re-read or written to.
- **Exciter bypassing Armature:** Exciter delegates to `createHookRegistry()`, never reimplements hook validation or wiring logic. The hook engine stays in Armature.
- **Direct settings.json mutation:** Never read-modify-write settings.json without atomicity. Use Lathe's `writeFileAtomic()` pattern to prevent corruption.
- **CLAUDE.md section clobbering:** Never write CLAUDE.md without first reading it and preserving all sections. A module that overwrites the entire file destroys other modules' sections.
- **Transcript path hardcoding:** The `transcript_path` is session-specific and its location pattern (`~/.claude/projects/[project_id]/[session_id].jsonl`) should never be assumed. Always use the path provided by the hook payload.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSONL parsing | Custom line-by-line JSON parser | `JSON.parse()` per line with error boundaries | Standard JSON per line, no special parser needed. The complexity is in format versioning, not parsing. |
| Atomic file writes | tmp file + rename logic | `lathe.writeFileAtomic(path, content)` | Lathe already implements this correctly with cleanup-on-failure. |
| Hook validation | Custom hook type checking | `HOOK_SCHEMAS` from `core/armature/hooks.cjs` | Single source of truth for valid hook types. |
| Provider contract | Custom validation for Lithograph API | `validateDataProvider('lithograph', impl)` | Enforces DATA_PROVIDER_SHAPE conformance automatically. |
| Service contract | Custom validation for Exciter API | `createContract('exciter', shape, impl)` | Standard contract creation with method presence validation. |
| Event emission | Custom event dispatching | `switchboard.emit(eventName, payload)` | Switchboard is the platform's event bus. |

**Key insight:** Both new components follow established patterns (provider-contract for Lithograph, service lifecycle for Exciter, options-based DI for both). The novelty is in the domain logic (JSONL parsing, settings merging, section markers), not the platform integration.

## Common Pitfalls

### Pitfall 1: Transcript File Concurrent Access
**What goes wrong:** Lithograph reads/writes the transcript JSONL while Claude Code is actively appending to it. A write operation that replaces content could corrupt the file mid-append, or a read could get a partial line.
**Why it happens:** The transcript file is shared between Claude Code's process and hook handlers. Hook handlers run synchronously within the Claude Code lifecycle (the hook blocks until the handler returns), but there is no file locking protocol.
**How to avoid:**
1. Reads happen during hook execution (Claude Code is blocked waiting for the hook response) -- the file is stable at that moment.
2. Writes use the temp file + rename atomic pattern via Lathe. This ensures the file is either the old version or the new version, never a partial write.
3. Never hold a file handle open across hook invocations. Read, process, write, close.
**Warning signs:** Corrupted JSONL (incomplete JSON on last line), parse errors on read.

### Pitfall 2: Transcript Format Change Breaks Parser
**What goes wrong:** Claude Code updates the JSONL format (field renames, structural changes, new entry types). Lithograph's parser fails on the new format.
**Why it happens:** The transcript format is undocumented. D-02 anticipates this with versioned parsers.
**How to avoid:**
1. Parser version detection on every first-read (not cached across sessions).
2. Unknown format returns a descriptive error, never crashes.
3. Parser fallback: if versioned detection fails, attempt raw JSON.parse per line and return untyped entries.
4. Log format version on first successful parse for diagnostics.
**Warning signs:** `UNKNOWN_FORMAT` errors in Lithograph results. New entry types appearing as `null` in queries.

### Pitfall 3: Settings.json Merge Conflicts Between Scopes
**What goes wrong:** A module registers hooks in project settings.json while user settings.json has different hooks for the same events. The merge produces unexpected behavior -- hooks fire twice or the wrong matcher wins.
**Why it happens:** Claude Code merges settings from all levels (user, project, local, policy). Hooks from all levels are active simultaneously (they add, not replace). A module that writes to both scopes creates duplicate hook entries.
**How to avoid:**
1. Exciter writes to ONE scope per operation. The caller specifies `scope: 'project'` or `scope: 'user'`.
2. Exciter's `readSettings()` method can merge and report which scope each entry came from.
3. Registration deduplication: before writing a hook entry, check if an equivalent entry already exists at any scope.
**Warning signs:** Hooks firing twice per event. Unexpected additionalContext appearing in Claude Code's context.

### Pitfall 4: CLAUDE.md Section Markers in Content
**What goes wrong:** A module's section content contains text that looks like a section marker (e.g., a code example showing the marker syntax). The parser matches the content marker instead of the actual section boundary.
**Why it happens:** Naive string matching for `<!-- name:start -->` inside content.
**How to avoid:**
1. Use unique, unlikely-to-collide markers: `<!-- dynamo:section:name:start -->` with the `dynamo:section:` prefix.
2. Parse markers greedily: find the FIRST start marker and the LAST end marker for a given name.
3. Validate that start comes before end.
**Warning signs:** Section content truncated or containing other sections' content.

### Pitfall 5: Reverie Migration Breaks Existing Tests
**What goes wrong:** Changing `reverie.cjs` from direct `createHookRegistry()` calls to `exciter.registerHooks()` breaks existing hook-handler tests that mock the registry directly.
**Why it happens:** Tests may depend on the internal wiring path (how hooks get registered) rather than the behavior (hooks fire when events occur).
**How to avoid:**
1. Hook handler tests (`hook-handlers.test.js`) test the handlers themselves, not the registration. These should be unaffected -- they test `handleSessionStart(payload)` directly.
2. The integration test (if any) that tests "hooks fire on Switchboard events" needs updating to go through Exciter instead of direct registry.
3. Run all existing Reverie tests BEFORE and AFTER migration. Diff test results.
**Warning signs:** Tests that mock `createHookRegistry` or check `registry.register()` call counts.

### Pitfall 6: Hook Registration Ordering with wireToSwitchboard
**What goes wrong:** Exciter's `registerHooks()` calls `wireToSwitchboard()` after registering each module's hooks. If multiple modules register hooks, calling `wireToSwitchboard()` multiple times could create duplicate event listeners on Switchboard.
**Why it happens:** `createHookRegistry()` creates a new registry per call. If Exciter creates a fresh registry per module, each wiring only wires that module's hooks. But if Exciter reuses one registry, multiple `wireToSwitchboard()` calls wire the accumulated listeners each time.
**How to avoid:**
1. Exciter creates ONE hook registry internally (in `init()`).
2. `registerHooks()` adds to that registry.
3. `wireToSwitchboard()` is called ONCE, after all modules have registered (during Exciter's `start()` phase, not during registration).
4. If late registration is needed, add a `rewire()` method that removes old listeners and re-wires.
**Warning signs:** Event handlers firing N times where N = number of modules registered.

```javascript
async function writeFileAtomic(filePath, content) {
  const tmpPath = filePath + '.tmp';
  try {
    fs.mkdirSync(path.dirname(filePath), { recursive: true });
    await Bun.write(tmpPath, content);
    fs.renameSync(tmpPath, filePath);
    return ok(undefined);
  } catch (e) {
    try { fs.unlinkSync(tmpPath); } catch (_) {}
    return err('WRITE_FAILED', `Atomic write failed: ${e.message}`, { path: filePath });
  }
}
```

## Transcript JSONL Structure (Verified via Research)

Each line in the transcript JSONL is a JSON object representing a conversation event:

- **User turns:** `{ "role": "user", "content": [...] }`
- **Assistant turns:** `{ "role": "assistant", "content": [...] }`
- **Tool use blocks:** Nested in content arrays with `tool_use_id`, `type: "tool_use"`, `input: {...}`
- **Tool results:** Nested with `type: "tool_result"`, `tool_use_id`, `content`

Key fields for Lithograph queries:
- `tool_use_id` -- unique identifier for correlating tool use with tool result
- `role` -- "user" or "assistant"
- `content[].type` -- "text", "tool_use", "tool_result"
- `content[].input` -- tool input parameters (the field claude-mem clears with `{}`)

## Open Questions

1. **Transcript JSONL exact schema**
   - What we know: Basic structure from claude-mem analysis (role, content array, tool_use_id).
   - What's unclear: Exact field set per entry type, metadata fields, timestamp format (if any), how compacted transcripts differ structurally.
   - Recommendation: Lithograph's `init()` or first `read()` should log the structure of the first 3 lines for diagnostics. The versioned parser handles structural surprises gracefully.

2. **Hook registration timing within lifecycle boot**
   - What we know: Lifecycle boots services in topological order. Exciter needs Switchboard (for wiring) and Lathe (for file ops). Modules register after boot via Circuit.
   - What's unclear: Whether Exciter's `wireToSwitchboard()` should happen in `start()` or be deferred until all modules have registered hooks.
   - Recommendation: Wire during `start()` for initial registrations. Provide a `rewire()` method for late-bound registrations (when a module loads dynamically).

3. **Exciter dependency registration in core.cjs**
   - What we know: core.cjs registers all services and providers. Exciter needs switchboard, lathe, and possibly armature hooks.
   - What's unclear: Exact dependency list for container registration.
   - Recommendation: Register Exciter with deps on `services.switchboard` and `services.lathe`. Armature hooks are imported directly (factory function, not a service).
