## Lithograph Scope

### Read/Write Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Full read/write | Ship read AND write operations with rollback. Matches PLT-02. Research validates approach. | ✓ |
| Read-only first | Only read/query operations. Add write later. Reduces risk but delays capability. | |
| Read + guarded write | Full read, writes require opt-in config flag. Safety net approach. | |

**User's choice:** Full read/write
**Notes:** Matches PLT-02 requirement exactly. Write ops needed for context sculpting in Phase 10+.

### Format Risk Handling

| Option | Description | Selected |
|--------|-------------|----------|
| Versioned parser | Abstract parsing behind version-detected parser. Add new parsers if format changes. | ✓ |
| Accept the risk | Parse known structure directly. Fix if it breaks. YAGNI argument. | |
| Schema validation layer | Zod-validate every JSONL line. Fail gracefully on unknown shapes. | |

**User's choice:** Versioned parser
**Notes:** Defensive but not paranoid.

### Path Source

| Option | Description | Selected |
|--------|-------------|----------|
| Session-scoped injection | Receive transcript_path once per session. Cache internally. | ✓ |
| Per-operation parameter | Every method takes transcript_path. More flexible, more ceremony. | |
| Config-based discovery | Discover from Claude Code session directory. No hook dependency. | |

**User's choice:** Session-scoped injection
**Notes:** Matches how providers work — initialized with a data source, then operate.

### Sync/Async Model

| Option | Description | Selected |
|--------|-------------|----------|
| Sync reads, async writes | Read synchronous (provider convention). Write async with atomic semantics. | ✓ |
| All synchronous | Match Journal's fully sync pattern. Risk: large reads block hooks. | |
| All async | Every op returns Promise. Breaks provider convention. | |

**User's choice:** Sync reads, async writes
**Notes:** Hooks need sync reads for fast response; writes can afford async.

## Exciter Boundary

### Ownership Model

| Option | Description | Selected |
|--------|-------------|----------|
| Own surface, delegate internals | Exciter owns Claude Code-facing files. Delegates to Armature for hook internals. Facade. | ✓ |
| Coordinate only | Convenience methods, no ownership. Weaker boundary. | |
| Own everything Claude Code | Absorb Armature's hook functionality. Requires refactoring M1 component. | |

**User's choice:** Own the surface, delegate internals
**Notes:** Exciter is the facade, Armature is the engine.

### Settings Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Both scopes | Project + user scope from day one. Exciter handles merge/precedence. | ✓ |
| Project scope only | Start with project scope. User scope is more sensitive. | |
| Scope-agnostic API | Methods take scope parameter. No opinion on which to use. | |

**User's choice:** Both scopes
**Notes:** PLT-03 explicitly requires both.

### CLAUDE.md Management

| Option | Description | Selected |
|--------|-------------|----------|
| Section-based ownership | Named sections with markers. Exciter manages create/update/remove. | ✓ |
| Append-only log | Modules append blocks. Simple but grows unbounded. | |
| Full file ownership | Exciter composes entire file from contributions. Clean but risky. | |

**User's choice:** Section-based ownership
**Notes:** Prevents clobbering between multiple consumers.

### Agent Definition Management

| Option | Description | Selected |
|--------|-------------|----------|
| Exciter writes agent files | Modules declare to Exciter. Exciter writes .md files. Single writer. | ✓ |
| Modules manage own files | Current pattern. Exciter provides registry only. Less control. | |
| Template-based generation | Structured definition → generated .md. Standardizes format. | |

**User's choice:** Exciter writes agent files
**Notes:** Single writer prevents conflicts.

## Provider vs Service

### Lithograph Classification

| Option | Description | Selected |
|--------|-------------|----------|
| Provider | Supplies and receives transcript data. Session path is injected config. | ✓ |
| Service | Transcript manipulation affects running session. Side-effect = service. | |
| Hybrid | Provider registration, service-like lifecycle hooks. Blurs boundary. | |

**User's choice:** Provider
**Notes:** Write-to-file is a provider operation, same as Journal.

### Exciter/Commutator Relationship

| Option | Description | Selected |
|--------|-------------|----------|
| Distinct domains | Commutator = I/O bus (routing). Exciter = integration surface (registration). | ✓ |
| Exciter absorbs hook routing | Move routing from Commutator to Exciter. Cleaner but refactors M1. | |
| Thin coordination layer | Tight coupling between Exciter and Commutator. | |

**User's choice:** Distinct domains
**Notes:** Commutator routes events; Exciter manages registrations. No overlap.

### Lithograph Directory

| Option | Description | Selected |
|--------|-------------|----------|
| core/providers/lithograph/ | Follows established provider directory pattern. Platform-level. | ✓ |
| modules/reverie/providers/ | Module-scoped. Limits other consumers. | |
| core/services/lithograph/ | Inconsistent with provider classification. | |

**User's choice:** core/providers/lithograph/
**Notes:** Registered in core bootstrap alongside Journal and Ledger.
