# Phase 9.1: Claude Code Integration Layer - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish the platform-level interface between Dynamo and Claude Code's native features. Two new components: Lithograph provider for transcript JSONL read/write/query, and Exciter service for managing the Claude Code integration surface (hook registration, agent definitions, skill definitions, settings.json, CLAUDE.md). Modules and extensions go through Exciter to implement Claude Code features, maintaining architectural integrity. Reverie's existing hook registration migrates to Exciter as the validation gate.

</domain>

<decisions>
## Implementation Decisions

### Lithograph Provider Scope
- **D-01:** Full read/write from day one. Lithograph ships with read (parse turns, query by type/tool_use_id/turn type/time range), write (replace block content, clear inputs), and rollback on failure. Matches PLT-02 requirement exactly. The research validates the transcript manipulation approach. Write ops are needed for context sculpting in Phase 10+.
- **D-02:** Versioned parser for undocumented transcript JSONL format. Inspect JSONL structure on first read, select matching parser version. If Claude Code changes the format, add a new parser without breaking existing code. Defensive but not paranoid.
- **D-03:** Session-scoped transcript_path injection. Lithograph receives transcript_path once per session (from SessionStart hook payload via Exciter/Commutator). Caches it internally. All subsequent operations use the cached path. Matches how providers work — initialized with a data source, then operate on it.
- **D-04:** Sync reads, async writes. Read/query operations are synchronous (matches Journal/Ledger provider convention). Write/manipulation operations are async with atomic semantics (temp file + rename). Hooks need sync reads for fast response; writes can afford to be async.

### Exciter Service Boundary
- **D-05:** Own the surface, delegate internals. Exciter OWNS the Claude Code-facing surface: settings.json files, agent .md files, skill definitions, CLAUDE.md content sections. It DELEGATES to Armature for hook registry internals. Modules call Exciter to register hooks (Exciter calls Armature underneath). Exciter is the facade, Armature is the engine. Exciter does not replace or absorb Armature's hook machinery.
- **D-06:** Both project and user scope settings.json management from day one. Project scope is `.claude/settings.json`, user scope is `~/.claude/settings.json`. Exciter handles merge/precedence logic. PLT-03 explicitly requires both scopes.
- **D-07:** Section-based CLAUDE.md ownership. Each module/extension claims a named section in CLAUDE.md, delimited by markers (e.g., `<!-- reverie:start -->...<!-- reverie:end -->`). Exciter manages sections: create, update, remove. No module can write outside its section. Prevents clobbering between multiple consumers.
- **D-08:** Exciter writes agent definition files. Modules declare agent definitions to Exciter (name, content, metadata). Exciter writes/updates/removes the .md files at `.claude/agents/`. Single writer prevents conflicts. Modules never touch the filesystem directly for agent definitions.

### Migration Strategy
- **D-09:** Exciter facade method for hook registration. Exciter exposes `registerHooks(moduleName, handlers)` which internally calls `createHookRegistry().register()` and `wireToSwitchboard()`. Reverie's `register()` changes from 10 lines of direct registry calls to 1 Exciter call. Armature `hooks.cjs` stays unchanged — Exciter is the facade. Existing Armature hook tests still pass; Reverie tests update the registration call.
- **D-10:** Migration happens IN Phase 9.1 alongside building Exciter. Proves Exciter works by migrating its first consumer (Reverie). This is the validation gate for the Exciter contract, as required by success criterion 3.

### Architecture Classification
- **D-11:** Lithograph is a provider, not a service. It supplies and receives transcript data. The session-scoped transcript_path is injected configuration, same as Journal's base directory or Ledger's database path. The "side effect" of modifying the context window is just a write operation on a file — Journal does the same thing.
- **D-12:** Exciter and Commutator have distinct domains. Commutator owns the I/O BUS — ingesting raw hook payloads, resolving event names, routing to Switchboard. Exciter owns the INTEGRATION SURFACE — what hooks exist, who registered them, what agents/skills are installed. Commutator routes events; Exciter manages registrations. They interact but don't overlap.
- **D-13:** Lithograph lives at `core/providers/lithograph/` following the established provider directory pattern. Registered in core bootstrap alongside Journal and Ledger. Any module or service can consume transcript data through it.

### Claude's Discretion
- Lithograph versioned parser implementation details (how to detect JSONL version, parser selection logic)
- Transcript JSONL query API method signatures beyond the functional requirements
- Exciter internal data structures for tracking registrations and installed components
- Settings.json merge/precedence algorithm details
- CLAUDE.md section marker format and parsing implementation
- Agent definition file template structure
- Exciter `registerHooks()` method signature beyond `(moduleName, handlers)` shape

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **Hook registry** (`core/armature/hooks.cjs`): Full registry with register(), wireToSwitchboard(), loadFromConfig(). Exciter delegates to this — wraps, does not replace.
- **Commutator** (`core/services/commutator/commutator.cjs`): HOOK_EVENT_MAP routes hook payloads to Switchboard events. Distinct domain from Exciter. No refactoring needed.
- **Provider contract** (`core/providers/provider-contract.cjs`): Base contract for providers. Lithograph conforms to this.
- **Journal provider** (`core/providers/journal/`): Reference implementation for flat-file provider pattern. Lithograph parallels this for JSONL.
- **Lathe service** (`core/services/lathe/`): Filesystem facade. Lithograph may use Lathe for atomic file writes.
- **lib/result.cjs**: ok/err pattern for all new components.
- **lib/contract.cjs**: createContract() for Lithograph and Exciter APIs.

### Established Patterns
- **Options-based DI**: All new components take injected dependencies.
- **Contract shapes**: SHAPE constant + createContract() for frozen public APIs.
- **Event emission**: Switchboard-based events on mutations (hook registrations, agent installs could emit events).
- **Provider init**: Providers initialize with injected configuration (paths, connections), then operate statelessly.
- **Service lifecycle**: Services go through init/start/stop via Armature lifecycle orchestrator.

### Integration Points
- **SessionStart -> Lithograph**: transcript_path from SessionStart hook payload injected into Lithograph for session-scoped caching.
- **Exciter -> Armature hooks**: Exciter's registerHooks() delegates to createHookRegistry().register() and wireToSwitchboard().
- **Exciter -> Lathe**: Exciter uses Lathe for file writes (settings.json, agent .md files, CLAUDE.md sections).
- **Reverie -> Exciter**: reverie.cjs migration — replace direct createHookRegistry() calls with exciter.registerHooks().
- **core.cjs -> Lithograph + Exciter**: Bootstrap registers both new components in the platform initialization sequence.

</code_context>

<specifics>
## Specific Ideas

### Lithograph as the Transcript Bridge
The research (09.1-RESEARCH-TRANSCRIPT-CONTROL.md) establishes that transcript_path is the key mechanism. Lithograph wraps this undocumented interface behind a versioned, validated provider contract. The versioned parser approach means if Claude Code changes the JSONL format, the fix is isolated to adding a new parser — not rewriting Lithograph consumers.

### Exciter as Architectural Enforcement
The core value is "everything routes through Dynamo." Exciter is the enforcement point for Claude Code integration. Without it, every module would independently manage settings.json, agent files, and CLAUDE.md — exactly the "bypass the patterns and paths Dynamo defines" anti-pattern. Exciter makes the integration surface a first-class platform concern.

### Section-Based CLAUDE.md
The marker-delimited section approach (`<!-- module:start -->...<!-- module:end -->`) enables multiple modules to coexist in CLAUDE.md without coordination. This is critical for when extensions layer on top of modules — each owns its section, Exciter arbitrates.

</specifics>

<deferred>
## Deferred Ideas

### Active Context Sculpting (Phase 10+)
With Lithograph's write capabilities established, the Context Manager can evolve from passive injection to active sculpting — replacing stale transcript entries with Self Model-framed reconstructions. Budget tracker becomes a controller, not just a monitor. Requires three-session architecture from Phase 10.

### Skill Definition Management
Exciter's contract includes skill definitions, but no current consumer needs this in Phase 9.1. Build the interface, implement when a concrete module needs to register skills.

### Hook Lifecycle Coordination
Exciter could orchestrate hook enable/disable at runtime (e.g., pause Reverie hooks during maintenance). Not needed for Phase 9.1 but architecturally natural for the integration surface owner.

</deferred>
