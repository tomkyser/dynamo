<objective>
Create the retroactive evaluator (LLM-driven re-evaluation of session fragments against the completed session arc) and the editorial pass (LLM-driven association index editorial operations: entity dedup, weight updates, domain boundary review, taxonomy narrative updates for domain merge decisions).

Purpose: REM-04 requires that fragments are re-evaluated retroactively after the full session is known -- early fragments may have different significance when viewed against the complete arc. REM-05 requires an editorial pass over the association index to maintain data quality: deduplicating near-synonym entities, adjusting association weights based on usage patterns, reviewing domain boundaries for merge candidates, and producing taxonomy narrative updates in Journal when domains are merged (per D-08). Both are LLM-driven editorial operations that run on Secondary (per D-06, D-08).

Output: Two REM sub-pipeline components with editorial prompt composition and result application.
</objective>

Retroactive evaluation per D-06:
- Mind re-reads all session fragments with full session summary context
- Updates: relevance scores (self_model_relevance), attention tags, pointers
- Session summary: available as input to evaluate()

Editorial pass per D-08:
- Entity dedup: find entities with high string similarity or identical referents
- Association weight updates: strengthen used associations, weaken unused
- Domain boundary review: find domain pairs with high entity overlap
- Taxonomy narrative updates in Journal for domain merge decisions: when domains are merged, `composeEditorialPrompt` includes instructions for the LLM to produce a brief narrative note describing the merge rationale and updated domain scope. The apply function writes this narrative to Journal via fragmentWriter as a consolidation-type fragment. This covers D-08's "taxonomy narrative updates in Journal" requirement. Full taxonomy governance (domain creation/split/retire/hard caps) remains Phase 12 scope per deferred FRG-07.

    `composeEvaluationPrompt(sessionSummary, fragments)` — builds a structured prompt:

    You are reviewing fragments from a completed session. Evaluate each fragment against the full session arc.
    For each fragment, decide: PROMOTE (to long-term memory) or DISCARD.
    For promoted fragments, update: relevance scores (0-1 for identity/relational/conditioning), new attention tags, updated pointers.

    Session Summary: {sessionSummary}

    Fragments to evaluate:

    `composeMetaRecallPrompt(recallEvents, sessionSummary)` — builds a prompt to create meta-recall fragments from significant recall events. Input: array of `{ query, fragments_composed, reconstruction_output, trigger, incorporated }`. Output prompt asks which recalls were significant enough to warrant meta-recall fragments and what their body text should be.

    `parseMetaRecallResponse(llmResponse)` — extracts meta-recall fragment specs from LLM response.

    `promoteFragment(fragment, evaluation, fragmentWriter, journal, wire)`:
    - Update fragment frontmatter with evaluation results (new relevance scores, attention tags)
    - Set `_lifecycle = 'active'`
    - Write updated fragment to active/ via fragmentWriter.writeFragment
    - Update Ledger fragment_decay.lifecycle via Wire write-intent
    - Delete working/ copy from Journal
    - Return ok/err Result

    `discardFragment(fragmentId, journal, wire)`:
    - Delete from Journal (working/ path)
    - Delete from Ledger tables (fragment_decay, fragment_domains, fragment_entities, etc.) via Wire write-intents
    - Return ok/err Result

    Factory `createRetroactiveEvaluator({ fragmentWriter, journal, wire, switchboard, config } = {})`:

    `evaluate(sessionSummary, fragments, recallEvents)`:
    - Compose evaluation prompt
    - Return `{ prompt: evaluationPrompt, metaRecallPrompt, apply: async function(llmEvalResponse, llmMetaRecallResponse) { ... } }`
    - The `apply` function processes LLM responses: parse decisions, promote/discard each fragment, create meta-recall fragments
    - Returns `{ promoted, discarded, meta_recalls_created, errors }`

    Note: The evaluator does NOT call the LLM directly. It composes prompts and provides an apply function. The full-rem.cjs orchestrator (Plan 05) will feed the prompts to Secondary's LLM context and pass responses back. This keeps the evaluator testable without LLM mocks.
