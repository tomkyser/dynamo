# Phase 3.2: Assay Federated Search - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver Assay, the federated search service that unifies query entry across all data providers. Assay executes provider-specific queries against Ledger (SQL) and Journal (markdown frontmatter), returning merged results with provider metadata identifying the source of each result. Assay does NOT own indexing in M1 -- it routes and merges. Indexing capabilities (Tika, Meilisearch, etc.) are deferred to M2+ when Reverie's recall requirements drive the need.

</domain>

<decisions>
## Implementation Decisions

### Provider Registration
- **D-01:** Options-based DI at init(). Providers injected via the options object, consistent with every other Dynamo service. Assay receives `{ ledger, journal, switchboard }` at init(). No auto-discovery, no registry lookup. Phase 4's framework (Armature) may add container-based resolution later, but Assay itself takes explicit provider references.
- **D-02:** Provider pool is extensible. Assay stores providers in an internal Map keyed by provider name. New providers can be registered after init via `registerProvider(name, provider)`. This supports Phase 4+ plugin providers without modifying Assay's core.

### Query Interface Design
- **D-03:** Dual query API. Two entry points:
  1. `search(query)` -- Federated search across all registered providers. Returns merged results with provider metadata on each result. This is the primary API that Reverie's recall pipeline will call.
  2. `searchProvider(name, query)` -- Targeted search against a specific named provider. Returns results from that provider only. For when a consumer knows exactly which provider to query (e.g., Tertiary scanning Ledger's association index only).
- **D-04:** Query object shape: `{ criteria, providers?, options? }`. `criteria` is the plain object matching the existing provider query(criteria) contract. `providers` is an optional array of provider names to restrict the federated search. `options` carries provider-specific hints (e.g., `{ sql: 'SELECT ...', limit: 50 }`).

### Provider-Specific Query Optimization
- **D-05:** Provider capability declaration. Each provider registered with Assay declares its query capabilities -- what kinds of queries it can optimize. Ledger declares SQL capability (it can execute raw SQL against DuckDB/SQLite). Journal declares frontmatter-scan capability (key-value matching against YAML headers). Assay routes the query to each provider using the most efficient method that provider supports.
- **D-06:** Capability-based routing. When a federated search arrives, Assay inspects the query and each provider's declared capabilities to determine the optimal query path:
  - If the query includes `options.sql` and the provider supports SQL -> pass the SQL directly
  - If the query includes criteria keys -> use the provider's standard query(criteria)
  - If the provider doesn't support the query type -> skip it (don't force lowest-common-denominator)

### Result Merging & Provider Metadata
- **D-07:** Provider metadata on every result. Each result returned from Assay includes `{ ...resultData, _provider: { name, type } }` so consumers know the source. This enables consumers to issue subsequent updates back to the correct provider -- a key requirement from the architecture plan.
- **D-08:** Provider-grouped result structure. Federated search returns `{ results: [...], providers: { ledger: { count, timing }, journal: { count, timing } } }`. Results are in a flat array (easier for consumers to iterate) but the providers summary gives per-provider diagnostics. Consumer (Reverie's Mind) owns ranking/scoring logic -- Assay does not impose relevance ordering across providers.

### Indexing Scope
- **D-09:** No indexing in M1. Assay in Phase 3.2 is a query federation layer only. The architecture plan mentions Tika, Meilisearch, and Opensearch -- these are M2+ concerns when Reverie's recall and sublimation volumes drive the need. Assay's pluggable provider model (D-02) makes adding index-backed providers straightforward later.

### Switchboard Integration
- **D-10:** Assay emits events for all operations: `search:started`, `search:completed`, `search:error`. Events include query shape, provider list, timing, and result counts. Enables platform-wide observability without coupling consumers to Assay's internals.

### Claude's Discretion
- Internal query plan optimization heuristics
- Provider capability schema (what fields, what format)
- Result array ordering within the flat list (provider order, insertion order, etc.)
- Timing measurement implementation (Date.now vs performance.now)
- Error handling for individual provider failures during federated search (skip vs fail-fast)
- Whether healthCheck() aggregates provider health or reports Assay's own status only

</decisions>

<specifics>
## Specific Ideas

- Architecture plan mentions "return the returning provider metadata so that subsequent updates may be issued after the fact for a given query" -- this is the core rationale for D-07. The _provider metadata must contain enough info to route a follow-up write/delete back to the correct provider.
- Reverie's Tertiary session does header-only scans (not full fragment retrieval) -- Assay's searchProvider() path to Ledger for association index queries is the fast path. Journal full-document queries are the slower recall path used by the Mind.
- The architecture plan's mention of Tika/Meilisearch/Opensearch maps to future providers that would register with Assay via registerProvider() -- the capability declaration model (D-05) makes this pluggable without modifying Assay core.

</specifics>

<deferred>
## Deferred Ideas

- **Full-text indexing** -- Tika, Meilisearch, or Opensearch integration for content-level search (architecture plan mentions these). Belongs in M2+ when recall volumes justify the infrastructure.
- **Relevance scoring/ranking** -- Cross-provider result ranking is a Reverie concern (Self Model relevance weighting). Assay returns results, Reverie ranks them.
- **Index management** -- Creation, maintenance, and optimization of search indexes. M2 concern driven by Reverie's fragment volume.
- **Caching layer** -- Query result caching for repeated searches. Premature in M1 without production usage patterns.

</deferred>
