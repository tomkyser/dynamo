# Phase 3.1: Wire Communication Service - Research

**Researched:** 2026-03-23
**Domain:** Inter-session MCP communication, transport abstraction, message routing, write coordination
**Confidence:** HIGH

## Summary

Wire is a core Dynamo service that enables concurrent Claude Code sessions to exchange typed messages through dual transport: Claude Code Channels API for native session event push, and an HTTP/WebSocket relay server (Bun.serve) for bulk data, high-volume streams, and fallback. Wire provides session registration with identity and capabilities, message routing with urgency-based prioritization, DuckDB single-writer coordination (Ledger write queue), backpressure management, and reconnection resilience.

The Channels PoC (`~/dev/cc-channels-poc/`) has validated the core pattern: an MCP server declares `claude/channel` capability, emits `notifications/claude/channel` events, and exposes MCP tools for session-to-session communication. The relay server (Bun.serve with HTTP + WebSocket) handles message routing with long-polling, WebSocket push, session registration, sequence numbering, and message buffering. The PoC used TypeScript/ESM -- Wire must be rewritten as CJS following the established service factory pattern. The PoC intentionally lacked urgency queuing, write coordination, batching, reconnection buffering, and transport abstraction -- all of which are Wire's core responsibilities.

**Primary recommendation:** Build Wire as five composable internal modules: (1) message envelope and protocol types, (2) transport abstraction layer with pluggable transports, (3) session registry with capability tracking and lifecycle events, (4) urgency-based message queue with backpressure, (5) Ledger write coordinator. The relay server is a separate Bun process managed by Conductor. The MCP channel server bridges transports to Claude Code sessions. All follow the established createWire() factory + createContract() pattern with options-based DI.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Dual transport, both active. Channels API for native Claude Code session event push. HTTP relay via Bun.serve (separate process managed by Conductor) for bulk data, high-volume streams, and fallback. Transport abstraction layer lets Wire route messages to the appropriate transport based on message characteristics.
- **D-02:** Channels maturity concerns overruled by user directive. This is a frontier project -- build on Channels directly, fix forward if issues arise. No defensive architecture that avoids Channels.
- **D-03:** Anticipate batching and high-volume/high-density message support. Tertiary sublimation stream can produce bursts. Wire must aggregate and batch rather than sending individually.
- **D-04:** Typed message envelope: `{ from, to, type, urgency, payload, timestamp, correlationId }`. Type is an enum (context-injection, directive, recall-product, sublimation, write-intent, snapshot, etc.). Urgency is an enum (background, active, directive, urgent) per the Reverie spec.
- **D-05:** Envelope enables routing, batching, and priority queuing by type and urgency. Wire inspects the envelope for routing decisions -- payload remains opaque to Wire.
- **D-06:** General-purpose topology. Wire supports arbitrary session registration with identity and capabilities. Reverie's Primary/Secondary/Tertiary is one configuration -- any module could define its own session patterns. No hardcoded roles in Wire.
- **D-07:** Registry pattern for session discovery. Sessions register on connect with identity + capabilities. Wire routes messages by registry lookup. Registry tracks who can write what (for write coordination). Registry emits lifecycle events (session:registered, session:lost).
- **D-08:** Relay runs as a separate Bun process managed by Conductor. Wire connects to the relay as a client. Better isolation -- relay survives Wire restarts, Conductor manages relay lifecycle (start/stop/health).
- **D-09:** Urgency-based queue with configurable depth limits. Separate queues per urgency level. Urgent messages bypass all queues. Background messages (sublimation) have bounded queue depth -- oldest dropped when full. Active/directive messages have larger bounds.
- **D-10:** Buffered reconnection with configurable TTL (default ~30s). Wire buffers messages for disconnected sessions. On reconnect, buffered messages delivered in order. After TTL expiry, session marked dead and buffered messages dropped. Registry emits 'session:lost' event for consumers to handle.
- **D-11:** Wire owns Ledger write coordination. Sessions send write intents via Wire messages (type: write-intent). Wire serializes writes to Ledger since DuckDB is single-writer.
- **D-12:** Priority FIFO with batching. Writes enter priority queue (urgent > directive > active > background). Within same priority, FIFO order. Wire batches compatible writes into single transactions where possible (e.g., multiple association index updates). Conflict resolution: last-writer-wins with version check.
- **D-13:** Dual API surface. Wire exposes MCP tools (send, subscribe, register) for Claude Code session-to-session communication via `@modelcontextprotocol/sdk`. Wire also exposes a native programmatic API for in-process consumers (Reverie module code). MCP for session-to-session, native for in-process.
- **D-14:** Wire declares `claude/channel` capability via MCP SDK. Channels API used for event push to Claude Code sessions. Matches the PoC pattern and official Channels contract.

### Pattern 1: Service Factory with Options-Based DI
**What:** Wire follows the established Dynamo service pattern -- factory function returns a validated, frozen contract via `createContract()`.
**When to use:** Always. This is the mandatory service pattern.
**Example:**

```javascript
  required: [

function createWire() {
  let _started = false;
  let _switchboard = null;
  let _conductor = null;
  let _registry = null;
  let _queue = null;
  let _writeCoordinator = null;
  let _transports = [];

  const impl = {
    init({ switchboard, conductor, ledger, configPath }) {
      _switchboard = switchboard;
      _conductor = conductor;
      // ... initialize internal modules
      return ok(undefined);
    },
    // ...
  };

  return createContract('wire', WIRE_SHAPE, impl);
}

module.exports = { createWire };
```

### Pattern 2: Transport Abstraction Layer
**What:** Pluggable transport interface so Wire can route messages through Channels API, HTTP relay, or future transports based on message characteristics.
**When to use:** All message send/receive operations go through the transport layer.
**Example:**
```javascript
// Source: D-01 decision + architecture plan
'use strict';

/**
 * Transport interface contract.
 * Each transport implements: send(envelope), connect(), disconnect(), healthCheck()
 * Transport selection is based on message characteristics:
 *   - Channels: session event push (directives, context injections)
 *   - Relay: bulk data, high-volume streams, batched sublimations
 */
const TRANSPORT_SHAPE = {
  required: ['send', 'connect', 'disconnect', 'healthCheck'],
  optional: ['sendBatch']
};

function createTransportRouter(transports) {
  function selectTransport(envelope) {
    // Urgent/directive -> Channels (lowest latency for session event push)
    // Background/batch -> Relay (handles volume, survives restarts)
    // Fallback: if Channels unavailable, use Relay for everything
    if (envelope.urgency === 'urgent' || envelope.urgency === 'directive') {
      const channels = transports.find(t => t.type === 'channels');
      if (channels && channels.isConnected()) return channels;
    }
    return transports.find(t => t.type === 'relay') || transports[0];
  }

  return { selectTransport };
}
```

### Pattern 3: Typed Message Envelope
**What:** All Wire messages use a typed envelope per D-04. Wire inspects envelope metadata for routing; payload is opaque.
**When to use:** Every message through Wire.
**Example:**
```javascript
// Source: D-04, D-05 decisions
'use strict';

const MESSAGE_TYPES = Object.freeze({
  CONTEXT_INJECTION: 'context-injection',
  DIRECTIVE: 'directive',
  RECALL_PRODUCT: 'recall-product',
  SUBLIMATION: 'sublimation',
  WRITE_INTENT: 'write-intent',
  SNAPSHOT: 'snapshot',
  HEARTBEAT: 'heartbeat',
  ACK: 'ack',
});

const URGENCY_LEVELS = Object.freeze({
  BACKGROUND: 'background',    // Sublimation, low-priority updates
  ACTIVE: 'active',            // Context injections, recall products
  DIRECTIVE: 'directive',      // Behavioral instructions
  URGENT: 'urgent',            // Immediate action required, bypasses queue
});

// Urgency ordering for priority queue (lower = higher priority)
const URGENCY_PRIORITY = Object.freeze({
  [URGENCY_LEVELS.URGENT]: 0,
  [URGENCY_LEVELS.DIRECTIVE]: 1,
  [URGENCY_LEVELS.ACTIVE]: 2,
  [URGENCY_LEVELS.BACKGROUND]: 3,
});

function createEnvelope({ from, to, type, urgency, payload, correlationId }) {
  return {
    id: crypto.randomUUID(),
    from,
    to,
    type,
    urgency: urgency || URGENCY_LEVELS.ACTIVE,
    payload,              // Opaque to Wire -- consumers define structure
    timestamp: new Date().toISOString(),
    correlationId: correlationId || null,
  };
}
```

### Anti-Patterns to Avoid
- **Hardcoding Reverie topology:** Wire must not know about Primary/Secondary/Tertiary. These are module-specific session roles. Wire sees sessions with identities and capabilities. Any hardcoded role assumptions violate D-06.
- **Synchronous message dispatch:** Wire must never block on message delivery. All transport operations are async. The urgency queue ensures ordering without blocking.
- **Direct Ledger access from sessions:** Sessions must not write to Ledger directly in a multi-session scenario. All writes go through Wire's write coordinator (D-11). This is a correctness requirement, not an optimization.
- **Tight coupling between transports:** Channels and relay transports must be fully independent. If Channels API changes, relay should be unaffected. Transport interface is the decoupling boundary.
- **Polling without backoff:** The relay client polling loop must implement exponential backoff on failures (PoC used fixed 2s delay -- Wire should do better).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MCP server protocol | Custom stdio JSON-RPC | `@modelcontextprotocol/sdk` Server class | Full MCP spec compliance, tool/resource handling, notification dispatch. ~20K lines of protocol code. |
| UUID generation | Custom ID generators | `crypto.randomUUID()` | Built into Bun/Node, RFC 4122 compliant, cryptographically random. |
| HTTP server | Custom TCP listener | `Bun.serve()` | Native HTTP + WebSocket with pub/sub. Handles connection lifecycle, upgrades, keep-alive. |
| JSON schema validation | Custom validators | `zod` schemas | Runtime type safety, composable, becomes mandatory with MCP SDK v2. |
| Event dispatch infrastructure | Custom pub/sub system | Switchboard (existing) + `node:events` | Switchboard already handles platform-wide event routing with wildcards, priority ordering. Wire emits through Switchboard for observability. |

**Key insight:** Wire's unique value is in the routing logic, urgency queuing, write coordination, and transport abstraction. The transport protocols themselves (MCP, HTTP, WebSocket) should use proven implementations.

## Common Pitfalls

### Pitfall 1: DuckDB Single-Writer Deadlock
**What goes wrong:** Multiple sessions attempt concurrent Ledger writes. DuckDB allows only one write transaction at a time. Second writer blocks, potentially timing out, or both sessions retry in a livelock.
**Why it happens:** Easy to forget the single-writer constraint when code runs in separate processes.
**How to avoid:** Wire's write coordinator (D-11, D-12) is the ONLY path for Ledger writes in a multi-session scenario. All sessions send `write-intent` messages through Wire. Wire serializes these into a priority FIFO queue and executes writes sequentially. This is architecturally critical, not an optimization.
**Warning signs:** "database is locked" errors, write timeouts, inconsistent state between sessions.

### Pitfall 2: Channels API Research Preview Instability
**What goes wrong:** Channels notification format, capability declaration, or event delivery semantics change between Claude Code releases. Wire's Channels transport breaks.
**Why it happens:** Channels are in research preview (as of v2.1.81). Anthropic explicitly states the protocol contract may change.
**How to avoid:** Transport abstraction layer (D-01). Channels transport is one pluggable module. If the API changes, only `channels-transport.cjs` needs updating. Relay transport provides full fallback capability. Per D-02: fix forward, do not build defensive architecture that avoids Channels.
**Warning signs:** `notifications/claude/channel` events silently not arriving, `<channel>` tags missing from Claude's context.

### Pitfall 3: Message Ordering Across Transports
**What goes wrong:** A directive sent via Channels arrives before a context-injection sent via relay. The directive references context that hasn't been delivered yet.
**Why it happens:** Dual transport means two independent delivery paths with different latencies. Channels is near-instant (stdio). Relay adds HTTP round-trip latency.
**How to avoid:** Use `correlationId` in envelopes to group related messages. For sequences that must be ordered, route them through the same transport. Document transport selection strategy: urgency-based by default, but consumers can force a specific transport via options.
**Warning signs:** Consumers receiving directives that reference unknown context, out-of-order message sequences.

### Pitfall 4: Reconnection Buffer Memory Exhaustion
**What goes wrong:** A session disconnects but its buffer keeps growing because other sessions continue sending messages to it. With 30s TTL and high-frequency sublimation stream, buffer can grow unbounded.
**Why it happens:** Background sublimation messages from Tertiary can fire every 5-10 seconds, each potentially multiple messages. 30s buffer at 2 msgs/sec = 60+ messages.
**How to avoid:** D-09 specifies bounded queue depth per urgency level. Background messages have the smallest bound. Oldest messages dropped when full (not newest -- the latest sublimation is more relevant). Configure depth limits via config, not hardcode.
**Warning signs:** Rising memory usage on relay server, slow message delivery after reconnection (large buffer flush).

### Pitfall 5: CJS Import Path for MCP SDK
**What goes wrong:** Using ESM import syntax or wrong require paths for `@modelcontextprotocol/sdk` in CJS files.
**Why it happens:** PoC used ESM (`import { Server } from '@modelcontextprotocol/sdk/server/index.js'`). Bun can `require()` ESM packages, but the correct CJS path matters.
**How to avoid:** Use `require('@modelcontextprotocol/sdk/server')` -- the package.json exports map routes this to `./dist/cjs/server/index.js`. Verified: the CJS build exists and exports `Server` class. The `StdioServerTransport` is at `require('@modelcontextprotocol/sdk/server/stdio')`.
**Warning signs:** `ERR_REQUIRE_ESM` errors, undefined exports, module resolution failures.

### Pitfall 6: Relay Server Process Orphaning
**What goes wrong:** Wire stops but the relay process keeps running. Port remains occupied. Next Wire start fails because port is in use.
**Why it happens:** Bun.spawn creates a child process. If the parent crashes without calling kill(), the child process is orphaned.
**How to avoid:** Conductor manages relay lifecycle. On Wire stop, Conductor sends SIGTERM to relay process. On Wire crash/ungraceful exit, Conductor's health check detects the orphaned relay and cleans it up. Relay should also bind to a configurable port and support `WIRE_RELAY_PORT` env var to avoid port conflicts.
**Warning signs:** EADDRINUSE on relay startup, multiple relay processes in process list, port conflicts.

### Urgency-Based Priority Queue
```javascript
// Source: D-09, D-12 decisions
'use strict';

const { URGENCY_PRIORITY, URGENCY_LEVELS } = require('./protocol.cjs');

function createPriorityQueue(config) {
  const _queues = {
    [URGENCY_LEVELS.URGENT]: [],      // No depth limit -- urgent always accepted
    [URGENCY_LEVELS.DIRECTIVE]: [],
    [URGENCY_LEVELS.ACTIVE]: [],
    [URGENCY_LEVELS.BACKGROUND]: [],
  };

  const _limits = {
    [URGENCY_LEVELS.URGENT]: Infinity,
    [URGENCY_LEVELS.DIRECTIVE]: config.directiveQueueDepth || 100,
    [URGENCY_LEVELS.ACTIVE]: config.activeQueueDepth || 200,
    [URGENCY_LEVELS.BACKGROUND]: config.backgroundQueueDepth || 50,
  };

  function enqueue(envelope) {
    const urgency = envelope.urgency || URGENCY_LEVELS.ACTIVE;
    const queue = _queues[urgency];
    const limit = _limits[urgency];

    if (queue.length >= limit) {
      // Drop oldest (not newest) -- latest message is more relevant
      queue.shift();
    }
    queue.push(envelope);
  }

  function dequeue() {
    // Dequeue in priority order: urgent first, background last
    for (const urgency of [
      URGENCY_LEVELS.URGENT,
      URGENCY_LEVELS.DIRECTIVE,
      URGENCY_LEVELS.ACTIVE,
      URGENCY_LEVELS.BACKGROUND,
    ]) {
      if (_queues[urgency].length > 0) {
        return _queues[urgency].shift();
      }
    }
    return null;
  }

  return { enqueue, dequeue, /* getDepth, isEmpty, flush */ };
}
```

### Session Registry Pattern
```javascript
// Source: D-06, D-07 decisions
'use strict';

const EventEmitter = require('node:events');

function createRegistry() {
  const _sessions = new Map();
  const _emitter = new EventEmitter();

  function register(sessionId, { identity, capabilities, writePermissions }) {
    _sessions.set(sessionId, {
      identity,
      capabilities: capabilities || [],
      writePermissions: writePermissions || [],
      connectedAt: Date.now(),
      lastSeen: Date.now(),
      status: 'active',
    });
    _emitter.emit('session:registered', { sessionId, identity, capabilities });
  }

  function unregister(sessionId) {
    const session = _sessions.get(sessionId);
    if (session) {
      _sessions.delete(sessionId);
      _emitter.emit('session:lost', { sessionId, identity: session.identity });
    }
  }

  function lookup(sessionId) {
    return _sessions.get(sessionId) || null;
  }

  function canWrite(sessionId, resource) {
    const session = _sessions.get(sessionId);
    if (!session) return false;
    return session.writePermissions.includes(resource) || session.writePermissions.includes('*');
  }

  return {
    register, unregister, lookup, canWrite,
    on: (event, fn) => _emitter.on(event, fn),
    off: (event, fn) => _emitter.off(event, fn),
    getSessions: () => Array.from(_sessions.entries()),
  };
}

module.exports = { createRegistry };
```

**Deprecated/outdated:**
- **PoC TypeScript/ESM approach:** Wire must be CJS. The PoC used TypeScript/ESM for speed of prototyping but Dynamo's source code is CJS only.
- **PoC flat message format:** The PoC used `{ from, to, content, meta }`. Wire uses the full typed envelope per D-04: `{ id, from, to, type, urgency, payload, timestamp, correlationId }`.
- **Single transport:** The PoC only had HTTP relay. Wire has dual transport with abstraction layer.

## Open Questions

1. **MCP SDK CJS `require()` with `async/await`**
   - What we know: MCP SDK's `Server.connect()` and `Server.notification()` are async. CJS supports top-level await in Bun but not in traditional Node.
   - What's unclear: Whether the channel-server.cjs should use an async IIFE wrapper or if Bun's CJS top-level await is reliable enough.
   - Recommendation: Use async IIFE wrapper `(async () => { ... })()` for safety. This pattern works everywhere and avoids reliance on Bun-specific behavior.

2. **Channels `--dangerously-load-development-channels` requirement**
   - What we know: During research preview, custom channels need the development flag. Wire's channel server is custom (not on Anthropic's allowlist).
   - What's unclear: When/whether Channels will exit research preview.
   - Recommendation: Document the flag requirement. Wire should work with or without it (relay transport provides full fallback). Per D-02: build on Channels directly, fix forward.

3. **Channel `meta` key naming constraints**
   - What we know: Meta keys must be identifiers (letters, digits, underscores). Keys with hyphens are silently dropped.
   - What's unclear: Whether this constraint will be relaxed in future Channels versions.
   - Recommendation: Use underscore_case for all meta keys in Wire envelopes: `from_session`, `urgency_level`, `message_type`. Never use hyphens.

4. **Write coordinator process affinity**
   - What we know: Wire's write coordinator serializes Ledger writes. Ledger (DuckDB) runs in-process.
   - What's unclear: If Wire's write coordinator should live in the relay process (central) or in each session's Wire service instance, given that DuckDB file-level locking prevents concurrent writes across processes.
   - Recommendation: Write coordinator lives in a single process that has exclusive Ledger access. Other sessions send write intents to this coordinator via Wire messages. The coordinator process is likely the Secondary session (which is the Mind and has the most writes) or a dedicated coordinator process.
