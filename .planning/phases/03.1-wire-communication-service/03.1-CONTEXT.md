# Phase 3.1: Wire Communication Service - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Build Wire, the inter-session communication service: an MCP server toolkit that enables concurrent Claude Code sessions to exchange typed messages via dual transport (Claude Code Channels API + HTTP/WebSocket relay). Wire provides session registration, message routing with urgency-based prioritization, Ledger write coordination (DuckDB single-writer constraint), backpressure management, and reconnection resilience. General-purpose toolkit — Reverie's three-session architecture is the first consumer, not the only configuration.

</domain>

<decisions>
## Implementation Decisions

### Transport Architecture
- **D-01:** Dual transport, both active. Channels API for native Claude Code session event push. HTTP relay via Bun.serve (separate process managed by Conductor) for bulk data, high-volume streams, and fallback. Transport abstraction layer lets Wire route messages to the appropriate transport based on message characteristics.
- **D-02:** Channels maturity concerns overruled by user directive. This is a frontier project — build on Channels directly, fix forward if issues arise. No defensive architecture that avoids Channels.
- **D-03:** Anticipate batching and high-volume/high-density message support. Tertiary sublimation stream can produce bursts. Wire must aggregate and batch rather than sending individually.

### Message Protocol
- **D-04:** Typed message envelope: `{ from, to, type, urgency, payload, timestamp, correlationId }`. Type is an enum (context-injection, directive, recall-product, sublimation, write-intent, snapshot, etc.). Urgency is an enum (background, active, directive, urgent) per the Reverie spec.
- **D-05:** Envelope enables routing, batching, and priority queuing by type and urgency. Wire inspects the envelope for routing decisions — payload remains opaque to Wire.

### Session Topology & Discovery
- **D-06:** General-purpose topology. Wire supports arbitrary session registration with identity and capabilities. Reverie's Primary/Secondary/Tertiary is one configuration — any module could define its own session patterns. No hardcoded roles in Wire.
- **D-07:** Registry pattern for session discovery. Sessions register on connect with identity + capabilities. Wire routes messages by registry lookup. Registry tracks who can write what (for write coordination). Registry emits lifecycle events (session:registered, session:lost).

### Relay Server Scope
- **D-08:** Relay runs as a separate Bun process managed by Conductor. Wire connects to the relay as a client. Better isolation — relay survives Wire restarts, Conductor manages relay lifecycle (start/stop/health).

### Backpressure & Flow Control
- **D-09:** Urgency-based queue with configurable depth limits. Separate queues per urgency level. Urgent messages bypass all queues. Background messages (sublimation) have bounded queue depth — oldest dropped when full. Active/directive messages have larger bounds.

### Reconnection & Resilience
- **D-10:** Buffered reconnection with configurable TTL (default ~30s). Wire buffers messages for disconnected sessions. On reconnect, buffered messages delivered in order. After TTL expiry, session marked dead and buffered messages dropped. Registry emits 'session:lost' event for consumers to handle.

### Write Queue (Ledger Coordination)
- **D-11:** Wire owns Ledger write coordination. Sessions send write intents via Wire messages (type: write-intent). Wire serializes writes to Ledger since DuckDB is single-writer.
- **D-12:** Priority FIFO with batching. Writes enter priority queue (urgent > directive > active > background). Within same priority, FIFO order. Wire batches compatible writes into single transactions where possible (e.g., multiple association index updates). Conflict resolution: last-writer-wins with version check.

### MCP SDK Integration
- **D-13:** Dual API surface. Wire exposes MCP tools (send, subscribe, register) for Claude Code session-to-session communication via `@modelcontextprotocol/sdk`. Wire also exposes a native programmatic API for in-process consumers (Reverie module code). MCP for session-to-session, native for in-process.
- **D-14:** Wire declares `claude/channel` capability via MCP SDK. Channels API used for event push to Claude Code sessions. Matches the PoC pattern and official Channels contract.

### Claude's Discretion
- Specific queue depth limits and TTL values (tunable via config)
- Relay server port selection and connection protocol details
- MCP tool naming and parameter schemas
- Internal message routing algorithm specifics
- Write batch grouping heuristics
- Session heartbeat/keepalive implementation details

</decisions>

## Existing Code Insights

### Reusable Assets
- `core/services/switchboard/switchboard.cjs` -- Event bus with action/filter dispatch. Wire should emit events through Switchboard for platform-wide observability (wire:message-sent, wire:session-registered, wire:write-queued, etc.)
- `core/services/conductor/conductor.cjs` -- Docker/process lifecycle management. Conductor will spawn and manage the relay server process.
- `lib/contract.cjs` -- createContract for service shape validation. Wire follows the same factory + contract pattern as all other services.
- `lib/result.cjs` -- Ok/Err result types used by all services.

### Established Patterns
- Service factory pattern: `createWire()` returns `createContract('wire', WIRE_SHAPE, impl)` — frozen, self-validated
- Options-based DI: `init({ switchboard, conductor, ledger, configPath })` for testability
- Lifecycle methods: init, start, stop, healthCheck (all services share this)
- Event emission via Switchboard on all significant operations

### Integration Points
- Wire → Conductor: relay process lifecycle management (spawn, stop, health check)
- Wire → Ledger: write coordination (serialized writes via priority queue)
- Wire → Switchboard: event emission for Wire operations
- Wire → MCP SDK: `@modelcontextprotocol/sdk` for MCP server contract and Channels capability
- Wire → Bun.serve: relay server (separate process)

</code_context>

<specifics>
## Specific Ideas

- User anticipates needing more than two transports eventually — Wire's transport abstraction should be pluggable, not hardcoded to Channels + HTTP relay
- DuckDB single-writer constraint is a first-class architectural concern, not an edge case — write coordination is core to Wire's value proposition
- The write queue serves double duty: serializing DuckDB access AND providing a coordination point for multi-session state management
- "Fix forward" approach to Channels issues rather than maintaining defensive fallbacks

</specifics>
